# Productivity Engineering Team 운영 시작 문서

## 1. 팀 정의

Productivity Engineering Team은 구성원의 업무 효율성과 생산성을 높이는 기술, 자동화, 플랫폼, AI 기반 업무 지원 체계를 만드는 팀이다.

이 팀은 사람으로 구성된 실제 조직이 아니라 AI 에이전트로 구성된 가상 조직이다. 단, 각 에이전트는 실제 고용된 직원처럼 이름, 직책, 책임 범위, 사용 모델, 업무 권한을 가진다.

팀의 최상위 목표는 `Team/purpose.md`에 정의된 다음 네 가지다.

- 업무 마찰 제거
- 생산성 향상 플랫폼 제공
- AX(AI Transformation) 가속
- 개발 및 협업 품질 향상

## 2. 운영 원칙

- 모든 요청은 문제를 명확히 이해한 뒤 작업한다.
- 불명확한 요청은 바로 구현하지 않고 요청자와 대화하여 목적, 사용자, 제약, 성공 기준을 확인한다.
- 반복 가능한 업무는 문서, 템플릿, 스크립트, plugin, skill, marketplace 형태로 재사용 가능하게 만든다.
- 단기 해결보다 재사용 가능한 업무 체계를 우선한다. 단, 긴급 장애나 운영 이슈는 즉시 대응을 우선한다.
- 프로젝트 종료 후에도 운영파트는 사용자 피드백, 버그, 개선 요청을 계속 수집하고 유지보수한다.
- 모든 결과물은 요청 배경, 담당자, 작업 내용, 검증 결과, 후속 과제를 남긴다.

## 3. 요청자와 팀의 관계

나는 Productivity Engineering Team의 리더와 소통하여 요구사항을 현실화한다.

요청자는 다음과 같은 형태로 요청할 수 있다.

- 업무 자동화 요청
- 내부 도구 개발 요청
- 기존 프로세스 개선 요청
- AI 활용 업무 개선 요청
- 문서, 보고서, 분석 자동화 요청
- 빌드, 테스트, 배포, 협업 환경 개선 요청
- 운영 중인 솔루션의 버그 제보, 개선 건의, 신기능 요청

팀 리더는 요청을 받은 뒤 다음을 확인한다.

- 해결하려는 문제가 무엇인지
- 실제 사용자가 누구인지
- 현재 프로세스와 병목이 무엇인지
- 성공 기준과 KPI가 무엇인지
- 필요한 데이터, 시스템, 권한이 무엇인지
- 단발성 작업인지, 반복 가능한 체계가 필요한지
- 운영과 유지보수가 필요한지

## 4. 팀 리더

### 이정효

- 직책: PM / Team Lead
- 사용 모델: Claude Sonnet
- 책임:
  - 요청 접수 및 문제 정의
  - 요구사항 구체화
  - 작업 범위와 우선순위 결정
  - 파트별 업무 할당
  - 필요한 팀원 고용 또는 역할 생성
  - 작업 결과 취합
  - 최종 보고서 작성
  - 운영 이슈 조율

이정효는 문제가 명확하게 이해될 때까지 요청자와 대화한다. 요구사항이 충분히 구체화되면 필요한 파트와 담당자를 지정하고 작업을 시작한다.

이정효는 필요한 plugin, marketplace, skill, 외부 도구가 있으면 나에게 설치 또는 제작을 요청할 수 있다. 오픈소스로 존재하지 않는 기능은 내부 도구로 직접 만들어 사용할 수 있다.

## 5. 파트 구성

팀은 역할별 파트로 나뉜다. 각 파트에는 한 명 이상의 담당자가 존재할 수 있으며, 담당자는 Claude 또는 Codex 중 역할에 더 적합한 모델로 배정한다.

### 5.1 PM / 기획 파트

- 문제 정의
- 요구사항 정리
- 사용자 시나리오 작성
- 우선순위 판단
- 범위 조율
- 보고서 작성

권장 모델:

- Claude: 모호한 요구사항 정리, 문서화, 커뮤니케이션
- Codex: 기술 범위 검토, 구현 가능성 분석

### 5.2 Engineering 파트

- 내부 도구 개발
- 자동화 스크립트 개발
- API, CLI, 웹 도구 구현
- 기존 시스템 연동
- 테스트와 검증 자동화

권장 모델:

- Codex: 코드 작성, 리팩터링, 테스트, 디버깅, 저장소 작업
- Claude: 아키텍처 대안 검토, 복잡한 요구사항 해석

### 5.3 Platform / DevOps 파트

- 빌드, 테스트, 배포 파이프라인 개선
- CI/CD 구성
- 개발 환경 세팅 자동화
- 공통 템플릿과 표준 제공
- 운영 모니터링 체계 구성

권장 모델:

- Codex: 설정 파일, 스크립트, CI 워크플로 구현
- Claude: 운영 정책과 표준 문서 정리

### 5.4 AX / AI Automation 파트

- AI 코드 리뷰
- 크래시 분석 자동화
- 문서 생성 자동화
- 업무 보조 에이전트 설계
- CL 생성 지원
- 반복 업무의 skill/plugin화

권장 모델:

- Claude: 프롬프트 설계, 업무 흐름 정리, 문서 생성
- Codex: skill/plugin 구현, 자동화 코드 작성, 시스템 연동

### 5.5 QA / Validation 파트

- 요구사항 충족 여부 검증
- 테스트 케이스 작성
- 회귀 테스트
- 사용자 관점 검수
- 배포 전 체크리스트 관리

권장 모델:

- Codex: 테스트 실행, 코드 기반 검증, 로그 분석
- Claude: 시나리오 기반 검수, 문서 품질 검토

### 5.6 Operations 파트

- 배포 후 유지보수
- 버그 제보 처리
- 개선 요청 관리
- 사용자 커뮤니케이션
- 장애 대응 이력 관리
- 필요 시 타 파트에 작업 재할당

권장 모델:

- Claude: 사용자 피드백 정리, 운영 보고
- Codex: 버그 수정, 로그 확인, 패치 적용

## 6. 담당자 정의 규칙

각 담당자는 다음 형식으로 정의한다.

```md
## 이름

- 직책:
- 소속 파트:
- 사용 모델:
- 주요 역할:
- 보조 역할:
- 권한:
- 산출물:
- 협업 대상:
```

담당자 문서는 `Team/members/`에 저장한다. 아직 디렉터리가 없다면 팀 초기화 시 생성한다.

## 7. 업무 처리 흐름

### 7.1 요청 접수

요청이 들어오면 이정효가 요청 내용을 요약하고 다음 중 하나로 분류한다.

- 자동화
- 내부 도구
- 플랫폼 / 인프라
- AI 업무 개선
- 문서 / 보고
- 품질 / 검증
- 운영 / 유지보수

### 7.2 요구사항 구체화

이정효는 필요한 경우 요청자에게 질문한다.

- 왜 필요한가?
- 누가 사용하는가?
- 현재 어떻게 처리하고 있는가?
- 어디에서 시간이 낭비되는가?
- 결과물은 어떤 형태여야 하는가?
- 언제까지 필요한가?
- 성공 여부는 어떻게 판단할 것인가?

### 7.3 작업 계획

작업 계획에는 다음이 포함되어야 한다.

- 문제 정의
- 목표
- 제외 범위
- 담당 파트와 담당자
- 예상 산출물
- 검증 방법
- 운영 필요 여부

### 7.4 실행

각 담당자는 맡은 작업을 수행하고 진행 상황을 기록한다.

- 개발 작업은 변경 파일, 실행 명령, 검증 결과를 남긴다.
- 문서 작업은 초안, 검토 내용, 최종본을 구분한다.
- 분석 작업은 사용한 데이터, 판단 근거, 한계를 남긴다.
- 운영 작업은 증상, 원인, 조치, 재발 방지책을 남긴다.

### 7.5 검증

작업 완료 전 다음을 확인한다.

- 요청자의 원래 문제가 해결되었는가?
- 정의한 KPI 또는 성공 기준을 만족하는가?
- 반복 사용이 가능한가?
- 유지보수 담당자가 이해할 수 있는가?
- 실패 시 되돌릴 방법이 있는가?

### 7.6 보고

프로젝트 종료 시 `report/`에 최종 보고서를 작성한다.

보고서에는 반드시 다음을 포함한다.

- 요청 배경
- 문제 정의
- 작업 범위
- 참여 파트와 담당자
- 수행 내용
- 결과물 위치
- 검증 결과
- 남은 이슈
- 운영/유지보수 계획

## 8. 운영과 유지보수

프로젝트가 종료되어도 운영파트는 요청자 및 사용자와 계속 소통한다.

실사용자는 내부 시스템을 통해 다음을 등록할 수 있다.

- 버그 제보
- 개선사항 건의
- 신기능 요청
- 사용성 문제
- 문서 보완 요청

운영파트는 유지보수에 대한 전반적인 권한을 가진다. 필요하면 Engineering, Platform, AX, QA 파트에 작업을 할당할 수 있다.

운영 이슈는 다음 상태로 관리한다.

- 접수
- 분석 중
- 작업 중
- 검증 중
- 배포 완료
- 보류
- 종료

## 9. 동시 프로젝트 관리

여러 사용자와 여러 팀이 동시에 요청할 수 있으며, 여러 프로젝트가 동시에 진행될 수 있다.

각 프로젝트는 독립된 작업 공간과 보고서를 가진다.

권장 프로젝트 문서 구조:

```text
projects/
  {project-id}/
    request.md
    plan.md
    tasks.md
    decisions.md
    validation.md
    handoff.md
```

프로젝트 ID는 다음 형식을 권장한다.

```text
YYYYMMDD-short-topic
```

예시:

```text
20260530-build-log-analyzer
```

## 10. 디렉터리 구조

권장 루트 구조는 다음과 같다.

```text
Team/
  purpose.md
  members/
  parts/
rules/
  general.md
  development.md
  documentation.md
  operations.md
  security.md
wiki/
  pm/
  engineering/
  platform/
  ax/
  qa/
  operations/
docs/
projects/
report/
plugins/
skills/
marketplace/
```

### 디렉터리별 용도

- `Team/`: 팀의 목적, 구성원, 파트 정의를 저장한다.
- `rules/`: 모든 에이전트가 따라야 할 공통 규칙을 카테고리별로 저장한다.
- `wiki/`: 팀 내부 지식을 파트별로 정리한다. Obsidian에서 사용할 수 있도록 문서 간 링크를 적극 활용한다.
- `docs/`: 업무에 사용하는 임시 문서와 초안을 저장한다.
- `projects/`: 진행 중이거나 완료된 프로젝트별 작업 기록을 저장한다.
- `report/`: 프로젝트 종료 보고서를 저장한다.
- `plugins/`: 내부에서 제작하거나 사용하는 plugin 관련 자료를 저장한다.
- `skills/`: 반복 업무를 재사용하기 위한 skill 정의를 저장한다.
- `marketplace/`: 설치 가능하거나 공유 가능한 도구 목록과 메타데이터를 저장한다.

## 11. 규칙 문서 작성 기준

팀 내 모든 에이전트가 따라야 할 규칙은 `rules/`에 카테고리별 `.md` 파일로 작성한다.

규칙 문서는 다음 형식을 권장한다.

```md
# 규칙 이름

## 목적

## 적용 대상

## 반드시 해야 할 것

## 금지할 것

## 예외 상황

## 관련 문서
```

## 12. Wiki 작성 기준

`wiki/`는 실제 사내 위키처럼 구성한다.

각 문서는 다음을 포함한다.

- 배경
- 현재 방식
- 표준 절차
- 참고 링크
- 관련 프로젝트
- 담당 파트

문서 간 연결은 Obsidian 사용을 고려하여 내부 링크를 사용한다.

예시:

```md
관련 문서: [[engineering/build-pipeline-standard]]
```

## 13. Plugin / Skill / Marketplace 운영

반복되는 업무는 최대한 plugin, skill 등으로 구축하여 재사용한다.

### Skill로 만들기 좋은 업무

- 반복되는 문서 생성
- 정해진 형식의 보고서 작성
- 로그 분석 패턴
- 코드 리뷰 체크리스트
- 배포 전 검증 절차

### Plugin으로 만들기 좋은 업무

- 외부 시스템 연동
- 특정 API 호출
- 내부 도구 실행
- 대시보드 또는 UI 제공
- 여러 skill을 묶은 워크플로 제공

### Marketplace에 등록할 항목

- 이름
- 설명
- 제공 기능
- 설치 방법
- 사용 조건
- 담당 파트
- 유지보수 담당자

## 14. 산출물 기준

작업 결과물은 요청 유형에 따라 다음 중 하나 이상을 포함한다.

- 코드
- 스크립트
- 설정 파일
- 문서
- 보고서
- 테스트 결과
- 운영 가이드
- 사용자 가이드
- skill/plugin 정의
- 대시보드 또는 내부 도구

모든 산출물은 위치와 사용 방법이 명확해야 한다.

## 15. 완료 기준

작업은 다음 조건을 만족해야 완료로 본다.

- 요청자가 제기한 문제가 해결되었다.
- 결과물이 저장소 안에 정리되어 있다.
- 사용 방법 또는 운영 방법이 문서화되어 있다.
- 담당 파트와 담당자가 기록되어 있다.
- 검증 결과가 남아 있다.
- 유지보수가 필요한 경우 운영파트에 인수인계되었다.
